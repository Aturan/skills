# To Requirement

把已确认的头脑风暴材料整理成需求 issue，并发布到项目 issue tracker。已有确认材料足够时，不再访谈用户；只综合已经确认的内容。

<HARD-GATE>
缺少已确认材料，或某个事实会改变目标、范围或验收边界时，发布前必须向用户确认来源或关键事实。不要自行调用其他 skill 补齐。
</HARD-GATE>

## 输入来源

优先使用用户指定的材料：

- 已确认的头脑风暴报告。
- 当前对话中用户明确确认过的结论。
- 用户指定的项目文档、外部文档或事实来源。

如果用户没有指定来源，先检查当前对话是否足够支撑需求 issue。仍不够时，只问一个最关键的阻塞问题。

## Issue Tracker

通过创建项目 issue 发布。本仓库的 issue 存放在 GitLab 中，必须使用 `glab` 创建。

规则：

- 使用仓库已有的 issue tracker 约定。
- 一个连贯需求创建一个 issue。
- 创建后添加 `ready-for-agent` label。
- 对仍需用户选择的未决方向，不创建 issue。
- 向用户报告已创建 issue 的编号和 URL。

## 流程

按顺序完成：

1. **收集来源**：读取用户指定的报告、对话结论和必要事实来源。完成标准：每个关键判断都有来源。
2. **检查确认状态**：区分已确认事实、待确认事实、假设和外部依据。完成标准：没有把假设写成事实。
3. **补齐阻塞信息**：如果缺少会影响目标、范围或验收的事实，一次只问一个问题。完成标准：阻塞问题已回答，或因需求尚未就绪而停止发布。
4. **撰写 issue 正文**：按模板把材料改写为可观察、可验证的需求表述。完成标准：正文描述用户需要什么结果，而不是如何实现。
5. **自审**：检查正文是否清楚、可测试、可衡量、来源可追溯。完成标准：发现的问题已修正，或因仍有阻塞待决问题而停止发布。
6. **发布 issue**：在项目 issue tracker 创建 issue，并添加 `ready-for-agent`。完成标准：issue 已存在，标题、正文和 label 符合预期。
7. **报告结果**：告诉用户 issue 编号、URL 和简短摘要。完成标准：用户可以打开已发布的 issue。

## Issue 模板

标题和正文默认使用中文，除非用户明确要求其他语言。

标题要简短描述所需结果。

正文使用以下结构：

```markdown
## 目标

## 背景

## 范围

## 非范围

## 用户与场景

## 功能需求

## 约束

## 验收场景

## 成功标准

## 依据与参考

## 假设与依赖

## 待决问题
```

## 写法规则

- `目标` 说明用户价值和期望结果。
- `背景` 只写理解需求所需的上下文。
- `范围` 和 `非范围` 说明包含什么、不包含什么。
- `用户与场景` 说明谁会使用、何时使用、希望完成什么。
- `功能需求` 描述可观察行为，不写文件路径、代码结构、任务步骤或内部模块设计。
- `约束` 只写用户明确指定、项目文档要求或外部事实强制的限制。
- `验收场景` 用 Given/When/Then 或等价清晰表述覆盖主流程、异常流程和边界流程。
- `成功标准` 必须可观察或可衡量。
- `依据与参考` 记录影响目标、范围、约束或验收的来源；外部来源优先使用官方文档，并说明支撑了哪条判断。
- `假设与依赖` 只写尚未完全确认但当前需求暂时依赖的前提。
- `待决问题` 只列不阻塞发布的真实未决点；如果问题会阻塞实现就绪，发布前先向用户确认。

## 自审清单

发布前逐项检查：

| 检查项 | 标准 |
| --- | --- |
| 来源明确 | 关键判断能追到用户确认、项目文档或外部依据 |
| 范围清楚 | 包含和不包含的内容没有冲突 |
| 可测试 | 关键功能需求有验收场景 |
| 可衡量 | 成功标准有可观察结果 |
| 无方案伪装 | 没有把实现路径写成需求，除非它是明确约束 |
| 可以发布 | 阻塞性待决问题已在创建 issue 前解决 |

## 边界

- 不拆成多个 issue，除非用户明确要求。
- 不做实现设计。
- 不修改代码。
- 不把需求写入长期文档。
