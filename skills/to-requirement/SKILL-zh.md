# To Requirement

把已确认的头脑风暴材料整理成需求 issue，并发布到项目 issue tracker。已有确认材料足够时，不再访谈用户；只综合已经确认的内容。

<HARD-GATE>
缺少已确认材料，或某个事实会改变目标、范围或验收边界时，发布前必须向用户确认来源或关键事实。不要自行调用其他 skill 补齐。
</HARD-GATE>

## 输入来源

优先使用用户指定的材料：

- 已确认的头脑风暴报告。
- 当前对话中用户明确确认过的结论。
- 用户指定的项目文档、外部文档或事实来源。

如果用户没有指定来源，先检查当前对话是否足够支撑需求 issue。仍不够时，只问一个最关键的阻塞问题。

## Issue Tracker

通过创建项目 issue 发布。本仓库的 issue 存放在 GitLab 中，必须使用 `glab` 创建。

规则：

- 使用仓库已有的 issue tracker 约定。
- 一个连贯需求创建一个 issue。
- 创建后添加 `ready-for-agent` label。
- 对仍需用户选择的未决方向，不创建 issue。
- 向用户报告已创建 issue 的编号和 URL。

## 流程

按顺序完成：

1. **收集来源**：读取用户指定的报告、对话结论和必要事实来源。完成标准：每个关键判断都有来源。
2. **检查确认状态**：区分已确认事实、待确认事实、假设和外部依据。完成标准：没有把假设写成事实。
3. **补齐阻塞信息**：如果缺少会影响目标、范围或验收的事实，一次只问一个问题。完成标准：阻塞问题已回答，或因需求尚未就绪而停止发布。
4. **撰写 issue 正文**：读取 `references/issue-artifacts.md`，按其中模板把材料改写为可观察、可验证的需求表述。完成标准：正文描述用户需要什么结果，而不是如何实现。
5. **自审**：使用 `references/issue-artifacts.md` 检查正文是否清楚、可测试、可衡量、来源可追溯。完成标准：发现的问题已修正，或因仍有阻塞待决问题而停止发布。
6. **发布 issue**：在项目 issue tracker 创建 issue，并添加 `ready-for-agent`。完成标准：issue 已存在，标题、正文和 label 符合预期。
7. **报告结果**：告诉用户 issue 编号、URL 和简短摘要。完成标准：用户可以打开已发布的 issue。

## 边界

- 不拆成多个 issue，除非用户明确要求。
- 不做实现设计。
- 不修改代码。
- 不把需求写入长期文档。
