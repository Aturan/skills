# README Maintainer

README 是门面和索引。它应该帮助读者理解项目或模块做什么、如何安全开始、权威细节在哪里。它不应该成为实现细节、长篇业务规则或算法说明的事实来源。

## 流程

1. 确认受影响的 README 和来源材料。
   完成标准：已经知道哪些 `README.md` 可能需要修改，以及哪些契约、规范、代码、脚本或工作流是权威来源。

2. 按公开相关性决定是否修改。
   完成标准：如果变更只影响私有实现，不改变用户可见行为、安装方式、API、配置、命令、文件布局或规范链接，保持 README 不变，并向用户说明这个决定。

3. 检查开头一句话。
   完成标准：第一段描述符合「功能主语 + 核心动作 + 解决的读者痛点」，不要用技术栈选择或实现细节开头。

4. 需要说明目录结构时，剪枝目录树。
   完成标准：目录树是 ASCII 认知地图，省略 `node_modules/`、缓存、生成文件、常规测试等噪音，并为每个保留项写出一个清晰职责说明。

   ```text
   src/middleware/  # 在受保护路由运行前校验 JWT
   ```

5. 同步公开契约。
   完成标准：示例只记录公开 API、CLI、安装方式、配置、输入、输出、错误和可观察副作用。契约有明确失败行为时，同时提供最小成功路径和最小失败路径。

6. 索引权威细节。
   完成标准：把长篇规则、配置大表、算法和深层实现说明替换或压缩为指向 living specs、API reference、设计文档或工作流文档的链接。除非当前 README 需要这些链接来完成导航，否则不要添加指向上游文档的回链，例如上层 README、`AGENTS.md` 或 `CLAUDE.md`。

7. 保守补丁。
   完成标准：尽量保留作者原有结构和语气，只修改受影响章节，并确保 Markdown 代码围栏、语言标记、命令、路径和链接正确。

## 规则

- 内部重构没有公开影响时，不要更新 README。
- 不要把大段规范内容复制进 README；改为链接到权威文档。
- 不要生成完整目录转储。只展示能帮助读者导航的条目。
- 对危险或意外的命令标注副作用，例如安装系统软件、删除数据、开放网络访问或改变外部状态。
- 优先沿用现有 README 的本地约定，不要强行套新模板。

## 汇报

完成 README 维护后，简要说明：

- 修改了哪些 README，或为什么没有修改。
- 一句话简介、目录树、契约示例和规范索引分别是已修改还是保持不变。
- 有哪些来源材料缺失或不确定。
