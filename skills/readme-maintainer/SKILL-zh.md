# README Maintainer

README 是门面和索引。它应该帮助读者理解项目做什么、如何安全开始、权威细节在哪里。它不应该成为实现细节、长篇业务规则或算法说明的事实来源。

## README 范围

- **仓库 README**：仓库的根入口。它负责仓库发现、安全起步、全局安装、全局命令、仓库约定、顶层结构、跨模块契约、跨模块导航，以及作用于整个仓库的高风险命令。
- **文档 README**：`docs/README.md` 是长期文档入口。它负责文档清单、文档导航，以及指向 ADR、PRD、设计文档、规范和 Agent 协作说明的链接。

本仓库只把根 `README.md` 和 `docs/README.md` 作为 README 入口。长期设计上下文进入 `docs/`、ADR 或 PRD；靠近实现的约束写进代码接口、类型、测试或必要注释；可重复执行流程进入 skill 或脚本。根 README 可以摘要这些事实，并链接到权威来源。

## 流程

1. 确认目标 `README.md` 和来源材料。
   完成标准：已经知道根 `README.md`、`docs/README.md`、两者或两者都不需要修改，以及哪些契约、规范、代码、脚本、文档或工作流是权威来源。

2. 应用 README 范围。
   完成标准：根 README 覆盖仓库发现、全局起步、全局约定、跨模块契约、顶层布局、文档导航和仓库级高风险命令；`docs/README.md` 只覆盖文档清单和文档导航。

3. 按公开相关性决定是否修改。
   完成标准：变更只影响私有实现时，保持 README 不变，并向用户说明这个决定。根 README 只在仓库入口、全局工作流、跨模块行为、顶层布局、文档导航、setup、命令或仓库级风险变化时更新。`docs/README.md` 只在文档清单、文档导航或权威文档位置变化时更新。

4. 检查开头一句话。
   完成标准：第一段描述符合「功能主语 + 核心动作 + 解决的读者痛点」，不要用技术栈选择或实现细节开头。

5. 需要说明目录结构时，剪枝目录树。
   完成标准：目录树是 ASCII 认知地图，省略 `node_modules/`、缓存、生成文件、常规测试等噪音，并为每个保留项写出一个清晰职责说明。目录树只展示顶层导航。

   ```text
   src/middleware/  # 在受保护路由运行前校验 JWT
   ```

6. 同步公开契约。
   完成标准：示例只记录公开 API、CLI、安装方式、配置、输入、输出、错误和可观察副作用。示例覆盖跨模块或全仓库行为。契约有明确失败行为时，同时提供最小成功路径和最小失败路径。

7. 索引权威细节。
   完成标准：把长篇规则、配置大表、算法和深层实现说明替换或压缩为指向 living specs、API reference、设计文档、工作流文档、脚本或 skill 的链接。除非当前 README 需要这些链接来完成导航，否则不要添加指向 `AGENTS.md` 或 `CLAUDE.md` 的回链。

8. 保守补丁。
   完成标准：尽量保留作者原有结构和语气，只修改受影响章节，并确保 Markdown 代码围栏、语言标记、命令、路径和链接正确。

## 规则

- 内部重构没有公开影响时，不要更新 README。
- 不要把大段规范内容复制进 README；改为链接到权威文档。
- 不要生成完整目录转储。只展示能帮助读者导航的条目。
- 不要创建模块级 `README.md` 文件来解释模块行为。
- 不要在 README 入口里重复模块详细用法；摘要说明并链接到 `docs/`、代码级契约、脚本或 skill。
- 对危险或意外的命令标注副作用，例如安装系统软件、删除数据、开放网络访问或改变外部状态。
- 优先沿用现有 README 的既有约定，不要强行套新模板。

## 汇报

完成 README 维护后，简要说明：

- 修改了哪个 README 入口，或为什么两个入口都保持不变。
- 一句话简介、目录树、契约示例和规范索引分别是已修改还是保持不变。
- 是否移动、摘要化或链接了重复事实。
- 有哪些来源材料缺失或不确定。
