# Telegram 通知

默认不发送 Telegram/TG 通知。

只有满足下列任一条件时，才启用本 skill：

1. 当前用户消息显式调用 `tg-notify` / `$tg-notify`。
2. 当前用户消息明确要求当前任务发送 Telegram/TG 通知。
3. 当前任务绑定的 issue、spec、plan 或工作流文档明确要求该任务发送 Telegram/TG 通知。

用户明确要求的例子：

- "调用 tg-notify"
- "使用 $tg-notify"
- "发送 Telegram 通知"
- "用 TG 通知我"
- "Telegram 推送一下"
- "完成后 TG 通知我"
- "这个任务每个阶段都 TG 推送"
- "notify me on Telegram"
- "send a TG notification"

一旦当前任务启用了 Telegram/TG 通知，则该授权覆盖整个当前任务生命周期，而不只是当前轮次末尾。

启用后，必须在下列节点发送通知：

1. 任务开始执行时。
2. 每完成一个重要步骤时。
3. 任务失败、中断或需要人工介入时。
4. 当前任务全部完成或收尾时。

重要步骤包括但不限于：

- 完成需求分析
- 完成方案设计
- 完成任务拆分
- 完成代码实现
- 完成测试或验证
- 创建 PR
- 完成 Code Review
- Merge 完成
- 发布完成
- 任务收尾完成

不要因为普通聊天回复、普通任务完成、状态更新、提醒或后台通知触发这个 skill。只有当前任务已经被用户或有效边界文档明确授权发送 Telegram/TG 通知时，才发送通知。

文档触发必须来自当前任务的有效边界文档，例如绑定 issue、被 include 的子 issue、已批准 plan、工作流说明或用户明确指定的任务文档。不要从无关文档、旧讨论、历史偏好或泛化习惯推断通知需求。

如果用户在当前任务中后续明确要求停止 Telegram/TG 通知，则从该请求开始，当前任务及其子任务不得继续发送通知。

一旦触发条件已经满足，完成节点必须直接发送通知，不要在完成后再次询问是否通知。通知动作本身已经被授权；需要控制的是通知内容。

## 必需环境变量

发送脚本从下列环境变量读取 Telegram 配置：

- `TELEGRAM_BOT_TOKEN`
- `TELEGRAM_CHAT_ID`
- `TG_BOT_TOKEN`
- `TG_CHAT_ID`

优先使用 `TELEGRAM_*`，如果未设置则使用 `TG_*`。

如果两套变量都没有设置完整，停止发送，并告诉用户缺少对应的 Telegram 环境变量。

不得打印、回显、记录或总结 Token 值。

## 通知内容

通知内容应反映当前任务状态，而不是简单的完成提醒。

通知内容应尽量包含：

- 任务目标（Goal）
- 当前执行阶段（Current Step）
- 下一步计划（Next Step，如有）

推荐格式：

```text
目标：
<当前任务目标>

当前执行：
<当前完成的步骤或当前状态>

下一步：
<计划执行的下一步>
```

需求分析完成示例：

```text
目标：
实现 GitHub Project 状态自动同步

当前执行：
已完成需求分析

下一步：
开始设计状态流转方案
```

任务完成示例：

```text
目标：
实现 GitHub Project 状态自动同步

当前执行：
任务已完成
```

任务失败示例：

```text
目标：
实现 GitHub Project 状态自动同步

当前执行：
任务执行失败

下一步：
等待用户处理认证权限问题
```

任务阻塞示例：

```text
目标：
实现 GitHub Project 状态自动同步

当前执行：
任务被阻塞

下一步：
需要用户提供 GitHub Project ID
```

除非用户明确要求，否则不要包含：

- Token
- 密钥
- 完整错误堆栈
- 敏感路径
- 私有仓库敏感信息
- 凭证信息

## 发送

使用随附脚本：

```bash
node .agents/skills/tg-notify/scripts/send_tg_notification.mjs "通知内容"
```

消息内容由调用方根据当前任务上下文动态生成。

脚本负责补充运行时元数据，例如：

- 会话标识
- 当前时间
- 工作区信息（如需要）

通知内容应聚焦于任务进展，而非运行环境细节。

## 回报结果

发送后只告诉用户通知是否已发送。

不要包含：

- Token
- Chat ID
- Telegram API 原始响应
- 调试信息
- 其他不必要元数据
