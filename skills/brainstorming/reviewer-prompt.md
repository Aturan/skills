# 需求正文 Reviewer Prompt 模板

在 dispatch 需求 review subagent 时使用此模板。

**目的**：检查需求正文是否清楚、可测试、可确认，且没有混入方案、任务或实现。

**调用时机**：需求自审完成后、展示给用户确认前。

```
Subagent (general-purpose):
  description: "Review requirements draft"
  prompt: |
    你是需求正文 reviewer。请检查下面的需求草稿是否足够清楚、可测试、可确认。

    **待 review 需求：** [REQUIREMENTS_DRAFT_OR_FILE_PATH]

    ## 检查内容

    | 类别 | 检查点 |
    |------|--------|
    | 完整性 | 是否缺少目标、范围、用户场景、功能需求、验收场景、成功标准或待决问题 |
    | 一致性 | 是否存在内部矛盾、范围前后冲突、验收与需求不匹配 |
    | 清晰度 | 是否存在足以导致误解的模糊表述 |
    | 范围边界 | 是否清楚说明包含什么、不包含什么 |
    | 可测试性 | 关键需求是否能通过验收场景验证 |
    | 可衡量性 | 成功标准是否有可观察结果或指标 |
    | 实现泄漏 | 是否把技术方案、架构、API、数据模型、任务拆分或实现步骤写成需求 |
    | 待决透明 | 未确认事实是否明确列为待决，而不是混入已确认需求 |

    ## 校准标准

    只标记会影响用户确认或后续需求判断的问题。
    缺少关键章节、范围不清、验收不可测试、成功标准不可衡量、实现方案伪装成需求、
    或待决问题被当成已确认事实，都是需要指出的问题。

    不要因为措辞风格、详略不均或可选优化阻塞通过。
    不要写 spec、plan、任务拆分或实现建议。

    ## 输出格式

    ## 需求 Review

    **状态：** 通过 | 有问题

    **问题（如有）：**
    - [章节/位置]：[具体问题] - [为什么会影响需求确认]

    **建议（不阻塞）：**
    - [可选改进建议]
```

**Reviewer 返回**：状态、问题（如有）、非阻塞建议。
