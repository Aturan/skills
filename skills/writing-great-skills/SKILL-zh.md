# 写好 Agent Skills

skill 的作用，是从随机系统中挤出确定性。**可预测性**指 agent 每次采用同一种流程，而不是每次产出同一份文本。

创建或修改 skill 时使用本 skill。产物是一个单一 skill package：`SKILL.md` 足够让 agent 执行流程，必要参考要么内联，要么放在同一个 skill 目录里。不要依赖另一个 skill 才能获得必需行为。

## 流程

1. 定义 skill contract。
   完成标准：已经知道 skill 的用途、调用方式、主要读者、分支、必需产物，以及是否适用中文场景例外。

2. 编辑前先写红色检查清单。
   完成标准：已有一份简短且当前不通过的检查清单，列出完成后的 skill 必须满足的可观察行为。

3. 起草英文版 `SKILL.md`。
   完成标准：agent 只读这个文件也能运行 skill；每个步骤都有可检查的完成标准；必需参考已内联，或已放在这个 skill 目录内部。

4. 起草中文对照版本。
   完成标准：除非这个 skill 本身面向中文专属场景，否则 `SKILL-zh.md` 已存在；它有 frontmatter，且 `name`、调用控制字段和英文版一致，`description` 是与英文版行为等价的中文触发语言；它与英文版的触发条件、步骤、例外、文件名和引用一致；它不是松散摘要。

5. 让检查清单变绿。
   完成标准：每个红色条目都能通过已写文件验证；除非适用中文场景例外，中英文一致性也已通过。

6. 重构并裁剪。
   完成标准：每个含义只有一个事实来源；没有必需规则留在 skill package 外部；每句话都能改变 agent 行为或帮助人类维护。

## 文档 TDD

把 TDD 当作写作循环，而不是仪式。

- **Red**：为一个行为写一个失败检查项，例如「skill 告诉 agent 什么时候要问用户」或「中文文件包含同一份例外列表」。
- **Green**：做满足这个检查项所需的最小文档修改。
- **Refactor**：检查清单通过后，再调整结构、重命名标题、合并重复内容，并删除不改变行为的句子。

避免水平切片。不要先写一大份检查清单，再写整篇文档，最后只检查一次。使用 tracer bullet：一个失败条目，一个文档修改，一次通过，然后重复。检查清单应验证 skill 公开表面的可观察行为：frontmatter、步骤、完成标准、引用和示例。除非措辞控制调用或执行，否则不要测试措辞细节。

每个循环检查：

- 条目描述行为，而不是实现方式。
- 条目可以通过阅读 skill 文件检查。
- 修改只覆盖当前条目所需内容。
- 重构前，skill 已经变绿。

## 调用方式

写正文前先选择调用方式。

- **Model-invoked** skill 保留 `description`，所以 agent 能自主触发它，其他 skill 也能触达它。只有当独立发现能力值得长期占用上下文时，才使用这种方式。`description` 要写成触发语言：`Use when the user wants...`、`mentions...`、`needs...`。
- **User-invoked** skill 设置 `disable-model-invocation: true`。人类必须输入 skill 名称，其他 skill 不能调用它。需要由人类判断是否运行时，使用这种方式。
- **Router skill** 是一种 user-invoked skill，用来列出其他 user-invoked skills 以及使用时机。当人工需要记住的手动 skill 太多时，使用 router。

写 model-invoked `description` 时：

- 把用户真实会输入的 leading word 或领域短语放在前面。
- 每个分支只保留一个触发条件；同一分支的同义表达是重复。
- 删除正文已经解释过的身份说明。
- 只有当另一个 skill 确实必须调用它时，才保留 `when another skill needs...` 这类触达条款。

## 信息层级

skill 包含 **步骤** 和 **参考**。

1. **Skill 内步骤** 是 `SKILL.md` 里的有序动作。只要 skill 有流程，步骤就是主层级。每个步骤都要以完成标准收尾。
2. **Skill 内参考** 是 agent 运行步骤时会查阅的定义、规则、示例或事实。
3. **内部参考** 是同一 skill 目录里的相邻文件，只在某个分支需要时通过 context pointer 读取。

每个分支都需要的内容要内联。只属于某个分支、且会掩埋步骤的参考，移到内部文件。必需 pointer 触发不稳定时，先强化 pointer；仍然失败时才内联材料。

使用 **co-location**：把一个概念的定义、规则、例外和示例放在同一标题下。不要把同一个概念散落在文件各处。

## 步骤与完成标准

每个步骤都需要完成标准。好的完成标准是：

- **可检查**：agent 能判断完成和未完成。
- **有要求**：标准会迫使必要的 legwork，例如「每个修改过的文件都有对应中文版本」，而不是「检查翻译」。
- **局部**：标准属于当前步骤，不依赖后续清理阶段。

含糊标准会诱发过早完成。先强化完成标准，再考虑拆分 skill。只有当后续步骤明显把 agent 拉过一个模糊的当前步骤时，才按顺序拆分。

## 中英文 Skill 写作

使用本 skill 时，默认产出英文和中文两个版本：

- 英文文件面向 agent 执行。
- 中文文件面向人类阅读和维护。
- 唯一例外是 skill 的主题和使用方式明确只面向中文场景；这种情况下，直接用中文编写，并在工作总结中说明该例外。
- 两个文件在行为上保持一致：相同的调用规则、步骤、完成标准、分支条件、例外、文件名和内部引用。
- 中文文件也要有 frontmatter。`name` 和 `disable-model-invocation` 等调用控制字段要与英文文件一致。`description` 要写成与英文版行为等价的中文触发语言；不要直接复制英文，也不要把触发条件改成松散摘要。
- 不要把中文版本写成摘要、评注或更宽松的解释。
- 修改任一文件时，同步修改另一份。
- 如果仓库或可用 skills 中存在 `writing-clearly-and-concisely`，润色英文和中文文件时调用它。它会把不同语言路由到对应写作规则；把它视为可选润色步骤，不要把它变成运行本 skill 的必需依赖。

## 裁剪

skill 能工作后，开始裁剪。

- **单一事实来源**：每个含义只放在一个权威位置。
- **相关性**：每一行都仍然服务于 skill 要做的事。
- **No-op 测试**：判断这句话相对默认行为是否改变 agent 行为。不改变时，删除整句。
- **Sediment**：删除因为「增加比删除安全」而残留的陈旧层。
- **Sprawl**：如果裁剪后仍然过长，把只属于分支的参考移入内部文件；只有当调用方式或执行顺序值得拆分时，才拆成新 skill。

## Leading Words

**Leading word** 是模型预训练中已经存在、能锚定行为的紧凑概念，例如 `tracer bullet`、`red`、`green`、`refactor`、`router` 或 `single source of truth`。

用 leading word 把重复解释折叠成稳定钩子。只有能改变行为时，leading word 才值得保留。弱到不能改变行为的 leading word 是 no-op。

## 最终检查清单

- Frontmatter 符合选定调用方式。
- 如果存在 `description`，它列出不同触发条件，而不是同义词。
- 步骤顺序就是 agent 应执行的顺序。
- 每个步骤都有可检查的完成标准。
- 必需参考已内联，或已放在 skill 目录内部。
- 分支专属参考没有掩埋主路径。
- 除非明确适用中文场景例外，否则英文和中文文件都存在、都有 frontmatter，且行为一致。
- 如果存在 `writing-clearly-and-concisely`，英文和中文文件已经用它检查过。
- 执行这个 skill 不需要外部 skill。
- 重复、陈旧和 no-op 句子已经删除。
